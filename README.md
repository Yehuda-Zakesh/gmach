# גמ"ח תוכנה — אפליקציית ווינדוס

גרסת Windows של ספריית התוכנה: **קובץ `.exe` בודד**, ולידו רק תיקיית `data`
אחת שנוצרת אוטומטית בהפעלה הראשונה. אין צורך לפרוס `index.html`, `css/` או
`js/` בנפרד — כל אלה ארוזים בתוך ה-exe עצמו.

## מבנה זמן-ריצה (אחרי בנייה)

```
📁 (תיקייה כלשהי, למשל כונן USB)
 ├── gmach.exe
 └── data/                    ← נוצרת אוטומטית בהפעלה הראשונה
     ├── database.json
     ├── software/            ← כאן מעתיקים תוכנות
     └── images/               ← אייקונים שהועלו מהניהול
```

## הפעלה

לחיצה כפולה על ה-exe פותחת את הקטלוג הציבורי. **"אפליקציה" ← "מסך ניהול"**
בתפריט העליון פותח את מסך הניהול (מוגן סיסמה, בדיוק כמו קודם) בחלון נפרד.

## מבנה המאגר (מקור)

```
app/            הפרונטאנד המקורי (index.html, admin.html, css/, js/) — נארז
                לתוך ה-exe בזמן build, ולא נשאר כקבצים חיצוניים
seed/           תוכן ברירת מחדל שנכתב לתיקיית data/ בהפעלה הראשונה בלבד
                (database.json ריק + שני קבצי README)
src-tauri/      קוד ה-Rust של המעטפת + הגדרות הבנייה
```

## בנייה

**אוטומטית (GitHub Actions):** ה-workflow ב-
`.github/workflows/build-windows.yml` בונה את ה-exe אוטומטית בכל push ל-main
(או ידנית דרך Actions ← Run workflow) על windows-latest, ומעלה אותו כ-artifact.

**מקומית על מחשב ווינדוס:**

```
# חד-פעמי: rustup.rs, ואז
cargo install tauri-cli --version "^2" --locked

cd src-tauri
cargo tauri build
```

ה-exe ייווצר ב-`src-tauri/target/release/*.exe`. יש להעביר אותו לתיקייה שבה
רוצים להריץ את האפליקציה (יחד עם ההרשאות המתאימות) — תיקיית `data/` תיווצר
שם אוטומטית בהפעלה הראשונה.

## מה שונה מגרסת הדפדפן המקורית

כל לוגיקת ה-UI (חיפוש, כרטיסים, עריכה, סינון) נשארה בדיוק כמו שהייתה. מה
שהוחלף הוא רק שכבת הגישה לקבצים, כי הדף כבר לא נטען ישירות מ-`file://` ליד
`data/`:

| פעולה | בגרסת הדפדפן (USB) | בגרסת ה-exe |
| --- | --- | --- |
| טעינת הקטלוג | `<script src="data/database.js">` | מוזרק ע"י Rust (`initialization_script`) לפני טעינת הדף |
| שמירה מהניהול | כתיבה ישירה (FSA) או הורדה + גרירה ידנית | פקודת `write_database` כותבת ישירות ל-`data/database.json` |
| העלאת אייקון | כתיבה ישירה (FSA) או הטמעה כ-`data:` URL | פקודת `write_image` כותבת ישירות ל-`data/images/` |
| הצגת אייקון בקטלוג | `<img src="images/...">` יחסי | `convertFileSrc` מול הנתיב המלא של `data/` |
| הורדת קובץ | `<a download>` | נפתח דיאלוג "שמור בשם" של המערכת (`download_item`/`download_package`); המשתמש בוחר יעד, והקובץ מועתק לשם בלבד — לא מגיע אוטומטית לשום תיקייה |
| פתיחת/הרצת קובץ | `window.open` בכרטיסייה חדשה | `open_item` מריץ את הקובץ עם התוכנה המשויכת אליו במערכת ההפעלה |
| הוספת תוכנה חדשה | גרירה ידנית לתיקייה + סריקה | כפתור "הוספת קבצים מהמחשב" (`import_software_files`) מעתיק ומזין ישירות לתהליך ההתאמה |
| סריקת תיקיית software | `<input webkitdirectory>` | **ללא שינוי** — זה כבר דיאלוג מערכת הפעלה אמיתי, לא תלוי בפרוטוקול הדף |

## נקודות שכדאי לבדוק בבנייה הראשונה

לא הייתה אפשרות להריץ `cargo build` בסביבה שהכינה את הקוד (אין שם Rust
toolchain כלל), אז יש כמה מקומות ספציפיים ב-API של Tauri v2 ששווה לשים לב
אליהם אם הבנייה נכשלת:

- `app.asset_protocol_scope().allow_directory(...)` — שם המתודה/חתימה
  המדויקים ב-Tauri v2 (ב-`main.rs`).
- `app.dialog().file().pick_files(...)` (עבור הוספת תוכנה) — ה-API של
  `tauri-plugin-dialog` v2 (callback vs. `async`/`await`), כולל `FilePath`
  שדורש `.into_path()` כדי להפוך ל-`PathBuf`.
- `dirs::download_dir()` — תלות חדשה לאיתור תיקיית ההורדות של המשתמש.
- `event.id().as_ref()` בהשוואת מזהה פריט התפריט.
- `WebviewUrl::App("index.html".into())` מול `admin.html` — ודא ששני הקבצים
  אכן נארזים מתוך `frontendDist` (`../app`).

אם משהו מזה לא מתקמפל, אפשר להעתיק לי את הודעת השגיאה המדויקת ואני אתקן.
